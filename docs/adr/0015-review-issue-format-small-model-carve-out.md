# ADR 0015 — ReviewIssueFormat uses a small model to produce structured snippets for GitHub issues

**Status:** Accepted  
**Date:** 2026-05-27

## Context

When a ReviewRun completes with `APPROVE`, the review agent emits `suggested_issues` — a list of `{hint, file, line}` triples pointing at non-blocking items worth tracking. The `hint` is a verbose, free-form sentence written by the main review model. When the user clicks "Create Issues" in the PR panel, Pawchestrator must turn each hint into a GitHub issue with a short title and a useful body.

The naive approach — use the hint verbatim as the issue title — produces issues that are too long to read, have no body, and are not actionable (see issues #176 and #177 for the motivating failure).

The standing policy (ADR 0002) prohibits LLM-generated text in GitHub artifacts except for Grill's questions output. Issues #176 and #177 were also created with an empty body (`with_generated_attribution("")`), so neither the title nor the body met any quality bar.

Three alternatives were considered:

1. **Ask the main review model to emit a pre-formatted `{title, body}` directly.** The main model already proved it writes verbose hints; changing the prompt shifts responsibility but does not enforce brevity. The main model is also expensive — spending output tokens on issue formatting is wasteful.

2. **Use a full agentic small-model call (with tool access) per issue.** Adds an agent loop, multiplies LLM round-trips, and risks the model going on an unbounded search. Inconsistent with the token-efficiency principle.

3. **Use a non-agentic small-model call per issue that returns constrained structured snippets; Python assembles the final issue body.** The model is a formatter, not an author. Output is maximally constrained. Python owns the prose structure. This is the same pattern as `criteria_dedupe`.

## Decision

The `issues` stage introduces a `review_issue_format` sub-step that runs one non-agentic small-model call per suggested issue before any GitHub issue is created.

**Model:** Claude Haiku (no reasoning) for ClaudeRunner; GPT-5.4-mini with `reasoning_effort = low` for CodexRunner. Same defaults as `criteria_dedupe`.

**Input per call:**
- The verbose `hint` string from the review artifact.
- The PR summary from the review artifact.
- The body of the matching `inline_comment` (looked up by `file` + `line`).
- A source snippet of ±15 lines around `line` from the local repo clone (omitted gracefully if no local clone is available).

**Output per call (structured JSON, no tool calls):**
```json
{
  "title": "≤80 char abbreviated title",
  "problem": "one sentence describing what is wrong",
  "acceptance_criteria": ["criterion 1", "criterion 2"]
}
```

**Body assembled by Python:**
```
**Where:** `path/to/file.py:123`

<problem>

## Acceptance Criteria
- [ ] criterion 1
- [ ] criterion 2

---
<sub>Generated by [Pawchestrator](https://github.com/LucaWichmann/Pawchestrator)</sub>
```

**Format calls run in parallel** (one per suggested issue, via `asyncio.gather`). **GitHub issue creation is sequential** (preserves safe DB write ordering, matches existing partial-progress behavior).

**Failure handling:** if any `review_issue_format` call fails or returns invalid JSON, the issues stage fails before creating any GitHub issues. The error is surfaced in the PR panel only — no PR comment is posted, no partial issues are created. The title is not truncated to a "best-effort" verbatim string because an issue with only a verbose title is practically useless for implementation.

**Title safety net:** Python hard-caps the model-supplied title at 256 characters (GitHub API limit) regardless of what the model returns.

## Why this is a carve-out from ADR 0002

ADR 0002's prohibition exists because "output tokens are expensive and run-status summaries add no value that structured state cannot provide more cheaply." Both conditions are addressed:

- **Token cost:** a non-reasoning small model (haiku / GPT-5.4-mini) with constrained structured output costs a fraction of a main-model call.
- **Value:** the issue title and body cannot be derived from structured state alone — the hint is intentionally terse and the body requires contextual interpretation of the source line.

Critically, the model does not write prose directly to GitHub. It emits **structured data snippets** (`title`, `problem`, `acceptance_criteria` items). Python owns the formatting, heading structure, `Where` injection, checkbox syntax, and attribution footer. The LLM contribution is bounded, predictable, and reviewable.

## Consequences

- `suggested_issues` schema in the review agent output changes from `list[str]` (titles) to `list[{"hint": str, "file": str, "line": int}]`. Each entry must reference an existing `inline_comment` by `file` + `line`; `normalize_review_artifact` validates this.
- New `review_issue_format` stage name registered in `runners.py` model-override logic (alongside `criteria_dedupe`).
- New `review_issues.py` module owns the format calls and GitHub issue creation (server.py `_create_review_issues` delegates to it).
- CONTEXT.md updated: `SuggestedIssues` definition, stage-to-runner table, review artifact schema, and comment policy carve-out note.
- ADR 0002 "Future actions" clause satisfied — this ADR is the explicit justification.
