# ADR 0023 — Smart routing: conditional plan skip via ScoutReport

## Status

Accepted (grilled 2026-06-04)

## Context

The Plan stage runs unconditionally on every pipeline run, regardless of issue complexity. Scout already classifies `next_recommended_stage` ("grill" | "plan" | "implement") and `risk` ("low" | "medium" | "high"), but pipeline ignores both fields.

For trivial, low-risk issues where the GitHub issue body already contains sufficient implementation context, the Plan stage produces marginal quality improvement at meaningful output token cost. Skipping Plan on these issues would save an entire LLM stage invocation.

`next_recommended_stage` is preserved in the ScoutReport schema specifically to enable this routing (see ADR 0022 Decision 3 note — the field was kept rather than dropped when it was found to be unused).

---

## Decisions

### Decision 1 — Opt-in smart routing config section

Smart routing is off by default. A new `[pipeline.smart_routing]` section in `config.toml` enables and controls it:

```toml
[pipeline.smart_routing]
enabled = false                   # opt-in; false = plan always runs
skip_plan_when = ["implement"]    # next_recommended_stage values that may trigger skip
require_readiness = ["ready"]     # readiness values required to allow skip
require_max_risk = "low"          # "low" | "medium" — ceiling risk level to allow skip
confirm_skip = false              # surface panel confirmation before skipping
```

All four sub-keys are independently configurable. A user wanting to skip plan on medium-risk issues sets `require_max_risk = "medium"`. A user wanting confirmation before skipping sets `confirm_skip = true`.

**Default `enabled = false`:** Plan stage consistently improves implementation quality. Smart routing is an explicit cost-reduction choice. Opt-in preserves the default full-quality pipeline.

---

### Decision 2 — Haiku micro-plan when plan is skipped

When smart routing fires, a lightweight plan is generated by Claude Haiku rather than running no plan at all. A thin plan improves implement agent output quality significantly compared to issue body alone, and a constrained Haiku call costs far less than a full Sonnet Plan stage.

**Micro-plan schema** (new `MicroPlan` skill, separate from `ImplementationPlan` for independent user override):

```json
{
  "approach_summary": "one sentence ≤100 chars",
  "steps": [
    "max 3 items — plain strings, each ≤15 words"
  ],
  "files_to_modify": ["list only"]
}
```

Stripped vs full plan: no `file_operations`, no `estimated_risk`, no per-step `notes`/`order`/`files_to_modify`. Haiku's role is orientation only: where + what, not how.

`file_operations` is excluded because it is used exclusively by the plan approval UI — it is not passed to the implement agent (confirmed in `_prompt_implementation_plan`).

**Alternative rejected:** No plan at all — implement receives only issue body + Scout findings. Rejected because real artifacts show Plan steps materially guide implement agent behavior. The cost of a constrained Haiku call is justified by the quality delta.

---

### Decision 3 — `confirm_skip` ties into plan approval gate

When `confirm_skip = true`, the micro-plan is surfaced to the user via the same plan approval panel UI before implement runs. The user can approve, reject (triggering a full Sonnet Plan stage instead), or abort.

When `confirm_skip = false`, the micro-plan is auto-approved and the pipeline proceeds immediately to implement.

This reuses the existing `PlanApprovalGate` infrastructure (`awaiting_plan_approval` status, approval event, panel rendering) without a new gate type.

---

## Consequences

- New `SmartRoutingSettings` dataclass under `PipelineSettings`.
- `run_pipeline` checks smart routing conditions after Scout and branches to micro-plan or full plan accordingly.
- New `MicroPlan` skill file at `pawchestrator/skills/MicroPlan/Skill.md`.
- New `run_micro_plan` function (thin wrapper, Haiku runner, constrained schema).
- `GET /config` endpoint gains `smart_routing` sub-object so the userscript can reflect routing decisions in the panel.
- `next_recommended_stage` in `ScoutReport` is now load-bearing — removing it would break smart routing.
